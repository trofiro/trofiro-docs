All communication happens over HTTPS, and every response is JSON-encoded, making the API straightforward to integrate with any language or toolchain.

## API Versioning

The current API version is **v1**, which is embedded directly in the URL path (e.g., `https://api.trofiro.com/v1/resources`). When Trofiro introduces breaking changes, a new version will be released under a new path segment (e.g., `/v2`). The previous version remains supported for a deprecation window announced in advance, giving you time to migrate. Non-breaking additions â€” such as new optional fields or new endpoints â€” are rolled out within the current version without a version bump.

## Request Format

Every request to the Trofiro API must be sent over HTTPS. Requests that include a body (such as `POST`, `PUT`, and `PATCH`) must encode that body as JSON and include the appropriate headers:

```bash
Content-Type: application/json
Accept: application/json
```

A minimal example request looks like this:

```bash
curl -X POST https://api.trofiro.com/v1/resources \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{"name": "My First Resource"}'
```

`GET` and `DELETE` requests do not require a body or `Content-Type` header, but including `Accept: application/json` is recommended for all requests.

## Response Format

All API responses return JSON. Successful responses wrap the primary payload in a `data` field alongside a `meta` object that carries request tracing information.

```json
{
  "data": { "..." },
  "meta": {
    "request_id": "req_abc123",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `data` | object \| array | The primary response payload |
| `meta.request_id` | string | A unique identifier for the request â€” include this when contacting support |
| `meta.timestamp` | string | ISO 8601 timestamp indicating when the response was generated |

## Pagination

Endpoints that return collections (such as `GET /resources` or `GET /users`) use **cursor-based pagination**. This approach is stable and efficient even as your data grows, because it anchors results to a specific position in the dataset rather than an offset.

Use the following query parameters to control pagination:

| Parameter | Type | Default | Max | Description |
|---|---|---|---|---|
| `limit` | integer | `20` | `100` | Number of records to return per page |
| `cursor` | string | â€” | â€” | Opaque cursor value from the previous response's `meta.next_cursor` |

Every paginated response includes the following fields inside `meta`:

| Field | Type | Description |
|---|---|---|
| `meta.next_cursor` | string \| null | Pass this value as `cursor` in your next request to retrieve the following page. `null` when there are no more pages. |
| `meta.has_more` | boolean | `true` if additional records exist beyond the current page |

**Example paginated request:**

```bash
curl "https://api.trofiro.com/v1/resources?limit=50&cursor=cur_01H8X2K9M3" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

To iterate through all pages, continue making requests with the `next_cursor` value until `has_more` is `false`.

## Rate Limits

Trofiro enforces per-minute rate limits that vary by plan. If you exceed your limit, the API returns a `429 Too Many Requests` response with a `rate_limit_exceeded` error code.

| Plan | Requests per minute |
|---|---|
| Free | 60 |
| Pro | 300 |
| Enterprise | Custom â€” contact your account manager |

Every API response includes the following headers so you can monitor your usage in real time:

| Header | Description |
|---|---|
| `X-RateLimit-Limit` | The maximum number of requests allowed in the current window |
| `X-RateLimit-Remaining` | The number of requests remaining in the current window |
| `X-RateLimit-Reset` | Unix timestamp (UTC) indicating when the current window resets |

When you receive a `429`, wait until `X-RateLimit-Reset` before retrying. For automated scripts, consider implementing exponential backoff to handle transient bursts gracefully.

## Error Format

When a request fails, the API returns a JSON object with an `error` field instead of `data`. The HTTP status code reflects the category of the failure.

```json
{
  "error": {
    "code": "resource_not_found",
    "message": "The requested resource could not be found.",
    "request_id": "req_xyz789"
  }
}
```

| Field | Type | Description |
|---|---|---|
| `error.code` | string | A machine-readable error code you can match programmatically |
| `error.message` | string | A human-readable description of the problem |
| `error.request_id` | string | Include this in support requests to help the team trace your specific call |

### Common Error Codes

| Code | HTTP Status | Description |
|---|---|---|
| `resource_not_found` | 404 | The requested resource does not exist or you do not have access to it |
| `validation_error` | 422 | The request body or query parameters failed validation; check `message` for details |
| `unauthorized` | 401 | No API key was provided, or the key is invalid or expired |
| `forbidden` | 403 | Your API key is valid but lacks the required scope for this action |
| `rate_limit_exceeded` | 429 | You have exceeded the request rate limit for your plan |
| `internal_error` | 500 | An unexpected error occurred on Trofiro's servers â€” retry with backoff and contact support if it persists |
